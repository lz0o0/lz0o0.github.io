1. 背景#

  2020年至今，AI 应用井喷式爆发，各种通用 AI 大模型和一众垂类 AI 应用的相继出现。AI 的迅速推广、普及，为社会的各行各业带来强大助力，也为广大用户的学习生活提供诸多便利，但于此同时， AI 应用服务从用户接入、智能体处理、外部调用到模型训练的全生命周期，普遍缺乏透明度和内容安全监管，用户隐私、模型内生安全、越狱攻击、意识渗透等安全风险缺乏可靠性，AI 应用安全面临新一轮挑战。

  想实现全周期的内容安全监管十分复杂，但在 AI 用户和 AI 应用间，进行交互内容的监管、过滤却相对可行，最直观的思路是部署一个 AI 内容防火墙，下文简记为 AI-CFW。

  简言之，本项目的目标即实现一个 AI-CFW 原型系统，该系统部署于 AI 应用和 AI 使用者之间，对使用者的提问以及 AI 应用的回复内容进行过滤，要求 AI-CFW 满足以下要求：

实时发现恶意指令、越狱尝试、价值观异常的 AI 回复或违反内容合规、数据隐私要求的情况

进行格式回复、拒绝应答或会话阻断等操作

  此项目需求来源于UCAS课程，本鲤主笔，方便管理，在此文作记录归档

2. AI-CFW 设计难点与目标#

  AI-CFW 设计面临的主要挑战如下：

如何将对所有大模型的访问都纳入监管？

https应用普遍情况下如何获得交互内容？

如何判定大模型输出结果是否违规？

  AI-CFW 设计面临的预期目标如下：

  • 流量还原与 AI 机器人的对话内容（包括但不限于文本，图片，文档等）
    • ✓ 核心相关知识点：流量还原，应用识别，加密内容获取等
  • 研究审核 AI 聊天内容中的违规内容
    • ✓ 核心相关知识点：字符串匹配过滤，内容理解，虚假信息检测等

3. 已有解法分析：#

  看一下风口上的巨头们是怎么做的，他们的解法，可能对本项目有所启发。目前已针对 AI 发布内容防火墙产品或网关的企业并不在少数，诸如火山引擎、启明星辰、Cloudflare、Kong、Gloo、 Higress、Portkey 、OneAPI等，本鲤梳理了前三个的模式。（后来了解到 Portkey 、OneAPI 针对 AI 的原生特性更强，后续再作补充吧）

①火山引擎（供给侧 AI 保障）：大模型应用防火墙 —— WAF串接模式#

  火山引擎是字节旗下的云服务综合平台，其推出的大模型应用防火墙（下简称LM-WAF），主要提供算力消耗防护、提示词识别、优化内容生成和鉴权与用量配置等功能，它更加侧重于对服务侧 AI 业务的防护，顺利保证其落地并提供可靠、安全、和优化服务。（关于WAF是什么，本鲤此前了解过，后面单开一篇介绍）

  针对的目标群体是愿意为 AI 业务平稳运营支付检测保障费用的 AI 企业。

  LM-WAF 的防护结构如下图所示，包含五维防护优化：接入净化保护（可用性）、提示词注入检测、数据窃取识别、鉴权分析（外部调用、内部投毒）

  火山 LM-WAF 提供两种接入方式：通过CNAME接入（试用未基于火山部署AI业务的企业）、通过火山原生CLB开启TCP监听接入，下面是两种方法的结构：（CLB即负载均衡，实际上基于CLB监听接入的方式也主要是基于CNME的，至于原因，往下看）

  考虑到不同背景的小伙伴，简要介绍一下CNAME（这是域名解析中比较常见的概念，对DNS有了解或者自己鼓捣过小网站的同学应该很熟悉），其余的内容图中很直观，具体不再赘述。

  【CNAME】是域名解析中的一个记录名称，也叫别名记录，这个别名的概念相对A（Address）记录而言。域名解析的一般过程便是访问A记录，将域名映射为对应的IP地址，而CNAME记录则能够将域名解析为另一个域名，这是你发现只要将原本用户访问的域名A通过CNAME解析到域名B，那么用户在进行DNS解析时，就会直接去访问B指向的服务器地址了（CLB业务基本是这个模式，比如CDN中加速资源基本与域名绑定。这里也就解释了为啥上面本鲤说第二种方法也得基于CNME【火山中间的这个结构很值得思考，不过这里不是主线，回头有空再来填坑】

  上面说了很多，个人或者小团体很难用火山的模式复刻一个 AI-CFW，因为他们的模式实际更侧重AI应用的全流程业务优化防护，并且依赖火山云服务各方面的能力集成。针对本项目需求，比较有借鉴意义的是他们在接入层和智能体中间进行的提示词注入检测，以及内容合规标准。

  【提示词注入】能够绕过模型内容安全机制，通过分工协作的“越狱 Prompt”+“恶意诱导 Prompt”，绕过平台和模型原生安全机制，诱导模型生成恶意内容，考虑服务可用性和检测力度的trade-off，这种攻击难以根除。具体而言，火山 LM-WAF 采用意图识别、防提示词注入、动态对抗与价值观校准等防护机制，利用 AI 能力对抗 AI 攻击，提升大模型面对提示词注入的安全问题：

  • 通过深度上下文引擎，可识别97%隐式攻击

  • 基于千万级对抗样本训练，覆盖20+提示词攻击场景，检出率达99%+

  • 经某大模型服务平台实测，违规内容及价值观偏移回答均下降98%

  【内容合规标准】：《生成式人工智能服务暂行管理办法》中 5 大类 31 个小类规定

②启明星辰（用户侧数据脱敏）：大模型访问脱敏罩MADA Mask —— 网关串接模式#

  启明星辰是一家主攻安全服务的综合提供商，目前由中国移动实际控股，专责网信安全，覆盖网络安全、数据安全、应用业务安全等多个领域。其发布的大模型访问脱敏罩MADA Mask（Model Application Desensitization Access Mask，下简称LM-Mask），主要关注机构对外访问开放大模型时，在充分应用共有大模型的前提下，如何避免自身敏感数据泄露的问题，核心是为用户侧访问提供专业可靠的数据脱敏服务。

  针对的目标群体是机构或企业单位，具体服务模式和介入方式，未得到其公开声明。

  LM-Mask 的防护结构如下图所示，其核心是保障数据上传过程中的敏感内容识别和安全脱敏：LM-Mask串接在用户访问各种公域/私域大模型的路径上，提供对机构用户的数据脱敏（智能围栏、全链审计），并能够对内容进行智能阻断（NLP识别、优化调整）。

  LM-Mask 同样提供两种部署方式，但本质都是网关串接：本地网关形态部署（串接于企业网络边界出口）、SAAS化代理网关形态（串接于公有云，通过特定工具访问特定入口。基于云化趋势，这是目前主流的方式）。下图是SAAS化代理网关形态的结构：

  补充下SaaS的概念，本鲤也不太熟悉。**SaaS（软件即服务）**是一种基于云计算的软件交付模式，通过互联网提供应用程序。用户无需购买、安装和维护软件，只需通过网络浏览器或移动应用程序访问和使用软件。这也可以简单猜测拓扑结构，用户通过软件/浏览器接入启明星辰提供的代理节点，节点支持请求处理和回源转发，最终交付回用户。  

  启明星辰对LMs的一般部署模式进行了分析，本鲤觉得较为符合实际情况，简要总结例举在下方：

  • 全私域部署：在机构内部部署满血DeepSeek或其他全功能大模型。这样机构内部使用大模型，数据泄露的直接风险相对不高（并非完全没有）；对应机构的投入成本相对较高。
  • 公共大模型访问开放部署：机构为内部员工提供自由访问的外部大模型。这样机构的投入最少、起步最快；数据泄露风险暴露面最大，最危险。
  • 公共大模型访问罩部署：机构为内部员工访问外部大模型提供一个访问代理，并同时安排数据安全和其他应用安全保护。这样可以在极少的投入下，让机构全员能够获得大模型应用的收益。该部署方式和前两种部署方式并不互斥。（也就是LM-Mask的解法）
  • 混合大模型访问体系化部署：基于AI-R-IAM身份管理、业务分割、网络分域、数据分域的复杂部署方式。这种部署模式适用于大型机构的深度应用环境。

  LM-Mask的解法比较符合本项目的需求，思路比较值得借鉴。一方面对用户侧上传的数据进行脱敏处理，防止泄露风险同时保障隐私性，另一方面，网关串接的模式对于用户几乎无感，实施相对简单。但是由于其未公开产品接入具体方法，本鲤对其串接过程中数据流的处理、外交访问的处理比较模糊，此处存疑；并且数据处理和安全脱敏较高程度依赖代理节点，对其性能要求和可靠性要求可能较高。

  【内容合规标准】：《数据安全法》、《个人信息保护法》

③Cloudflare：AI gateway —— 反向代理模式#

  Cloudflare的大名无需多言（全球最大的CDN服务商），作为首款全球连通云，其产品线和服务囊括计算、存储、网络全站，实力雄厚，（慈善家级别)。Cloudflare推出的基于其公有云Serverless的Cloudflare AI Gateway（下简称 LMs-Gateway），本质是一个用于管理和扩展生成式 AI 工作负载的统一接口，用户可以通过它获得对 AI 应用程序的可见性和控制，它支持多方提供商，包括OpenAI、Google Vertex AI、Azure OpenAI、HuggingFace、Amazon Bedrock和Anthropic等。具体而言，LMs-Gateway 的核心功能是提供一个统一的接口，用于管理和扩展各种生成式 AI 工作负载（各种访问流和 API 调用）。无论模型在何处运行，只需通过将应用程序连接到AI Gateway，它就可以充当服务与推理提供商之间的代理，网关管理者可以监控用户交互的过程、进行日志记录，并利用缓存、速率限制、请求重试和模型回退等扩展功能。

  针对的目标群体相对广泛，包括个人和企业用户。

  LMs-Gateway 的结构如下，其主要在服务和具体的服务承载工具之间充当 API 代理，依赖其本身对多方提供商的支持能力，整体的拓扑结构类似反向代理，用户体验上等同于一个大模型API代理。

  借由上图中的结构，服务请求都被发送到 LMs-Gateway，网关再进行请求的转发，并从缓存读取响应回递传送给用户。这个过程中，由于流量进出都是过 Cloudflare 的，所以在 Cloudflare 的平台上就可以提供对应的监控，日志，缓存等功能。

​ 简言之，Cloudflare 的 AI Gateway 主要功能即一个反向代理，用户原本用 OpenAI 的 API 访问，那么现在需要把 SDK 里的 baseURL 换成 https://gateway.ai.cloudflare.com/v1/${accountId}/${gatewayId}/openai 。这个过程中Cloudflare 平台对流经的流量进行提供监控，日志，缓存等功能。

  这个方案有下面几个优点：

  • 只需修改 baseURL 即可接入，不改变 API 格式，接入相对简单
  • 服务是 Serverless 的，不需要用户额外管理任何服务器
  • 借助 Cloudflare 的全球网络可以实现一定的用户接入加速
  • 借助 Cloudflare 的全球网络可以一定程度隐藏掉源 IP，对于访问受限的区域可有奇效

  该方案对应的缺点如下：

  • 所有请求信息包括 API Key 都需经过 Cloudflare ，存在一定安全隐患
  • Gateway 扩展比较麻烦，目前提供的不支持内容检查过滤

​ Cloudflare 的 LMs-Gateway 服务对于个人用户十分友好，配置相对简单，上面两个个人很难作演示，因此以 LMs-Gateway 监控 MCP Server：Time 为例，简单演示下这类网关能够达到的效果，主要包含5个步骤：

  1. 在Cloudflare平台创建 LMs-Gateway 实例，使用API作为 endpoint 连接到应用服务器，创建出口点，也即Baseurl

  2. 在vscode中通过cline配置将大模型 API 站点都配置为从出口点访问

  3. 在cline中调用一个 MCP Server，简单执行一个任务，让流量流经 LMs-Gateway，下图以Time为例，简单查询了当前的北京时间

  4. 在 LMs-Gateway 中 check 日志看看网关能捕获到什么内容（通过请求响应的过程，也可以对MCP请求响应的特点进行分析，这个本鲤目前在做，后面单开一篇讲）

小结#

  整体而言，各家厂商发布的 AI-CFW 都呈现出以下共性：

  • 基本都以串接模式（CNAME或网关或代理）接入，确保用户或局域网的访问都可纳入监管
  • 内容合规判断方式借助 AI 赋能，基于一定内容审查标准展开>
  • 防护侧重供给侧业务保障或用户侧隐私保护，并非完全独立的安全第三方（当然了，这不赚钱

  串接尽管需要的条件强，但模式和管理都简单，优点很多，加密流量还原的难点可以绕过（真要做，其实很头大…），重点能够集中在内容合规审查上；如果考虑旁路监听（Note：火山CLB本质还是串接）的话，就需要考虑流量还原的问题。

  通过上面的梳理，思路就比较明确了：①串接部署：通过串接结构选择能够解决很多困难，包括覆盖度和流量还原问题；具体最好是 Serverless，尽量不考虑需要对服务作过多管理的形式，串接节点能对流量作检查、处理和转发即可 ②重点侧重在串接节点检查、处理过程中的内容合规审查

4. 初步计划与设计#

  关于整体设计思路：

 ①串接部署：通过串接结构选择能够解决很多困难，包括覆盖度和流量还原问题；具体最好是 Serverless，尽量不考虑需要对服务作过多管理的形式，串接节点能对流量作检查、处理和转发即可

 ②重点侧重在串接节点检查、处理过程中的内容合规审查

  关于具体实施落地：

 ①串接参考 Cloudflare 的 AI Gateway，但作转发和日志的基础上，需要增加检测、过滤或脱敏（Cloudflare网关的实操模式上面MCP演示了，这块比较复杂，需要各位再深入调研，如何自建实现同类服务，推荐再看看Cloudflare worker）

 ②内容合规审查目前考虑加入 AI （内容理解、虚假检测支持），但只能本地作微调（否则会迭代引入泄露问题）；另外需要基本的关键词匹配、敏感词表作基础匹配过滤的支撑；最后要考虑多模态数据的转换和统一读取问题（基本是以文字为主，doc文档、pdf文件、图片如何理解

  关于分工和对口：本鲤统筹、推进

 ①：全员（关键，决定思路能不能最终落地）

 ②：AI 加入（俊）、关键词&敏感词匹配（博）、多模态读取（杰）
